## 2. Public-Data-Only Policy

### 2.1 Hard Boundary

All data ingested, processed, and displayed by this platform comes from **publicly accessible sources**. The platform:

- **DOES** aggregate public APIs (OpenSky, USGS, NWS, GDELT, NVD, GitHub)
- **DOES** display publicly accessible web cameras
- **DOES** search publicly indexed documents and repositories
- **DOES NOT** access private databases, email accounts, or protected systems
- **DOES NOT** perform credential stuffing, brute-force attacks, or social engineering
- **DOES NOT** intercept communications or perform man-in-the-middle attacks
- **DOES NOT** scrape data behind authentication walls without authorization

### 2.2 Source Verification

Every data source used by the platform is documented in `docs/APP_MAP.md` with:
- API endpoint URL
- Data type returned
- Rate limit compliance
- Terms of service adherence

## 3. Authentication & Authorization

### 3.1 Authentication Flow

```
User → Manus OAuth → JWT Session Cookie → Protected Procedures
```

- Authentication is handled by Manus OAuth (industry-standard OAuth 2.0)
- Session tokens are signed with `JWT_SECRET` (server-side only)
- Cookies are HttpOnly, Secure, SameSite=Strict
- No passwords are stored in the application database

### 3.2 Authorization Levels

| Level | Access | Enforcement |
|-------|--------|-------------|
| **Public** | Read-only OSINT data, search, browse | `publicProcedure` |
| **Authenticated** | Create cases, submit evidence, run playbooks | `protectedProcedure` (ctx.user required) |
| **Admin** | User management, system configuration | `adminProcedure` (role === 'admin') |

### 3.3 Role-Based Access Control

The `user` table includes a `role` field with values:
- `user` — Standard analyst access
- `admin` — Full platform administration

Role escalation requires direct database modification by a platform administrator.

## 4. Immutable Audit Trail

### 4.1 Design

The audit log implements a **blockchain-lite hash chain** using SHA-256:

```
Entry N:
  - id: auto-increment
  - action: string (e.g., "evidence.create", "playbook.execute")
  - actorId: user ID or "system"
  - details: JSON payload
  - timestamp: UTC milliseconds
  - prevHash: SHA-256 hash of Entry N-1
  - hash: SHA-256(id + action + actorId + details + timestamp + prevHash)
```

### 4.2 Properties

- **Append-only**: No UPDATE or DELETE operations on the audit_log table
- **Tamper-evident**: Any modification to a historical entry breaks the hash chain
- **Verifiable**: The entire chain can be validated by recomputing hashes from genesis
- **Non-repudiable**: Every action is attributed to a specific user or system process

### 4.3 Logged Actions

| Action Category | Events Logged |
|----------------|---------------|
| Authentication | Login, logout, session creation |
| Evidence | Create, update status, delete (soft) |
| Cases | Create, update, close, add evidence |
| Playbooks | Execute, complete, fail |
| Agents | Chat sessions, commands issued |
| Reports | Generate, export, download |
| Connectors | Connect, disconnect, health check |
| System | Configuration changes, errors |

### 4.4 Chain Verification

```typescript
// Verify audit chain integrity
async function verifyAuditChain(): Promise<{valid: boolean, brokenAt?: number}> {
  const entries = await db.select().from(auditLog).orderBy(auditLog.id);
  for (let i = 1; i < entries.length; i++) {
    const expected = sha256(
      entries[i].id + entries[i].action + entries[i].actorId +
      entries[i].details + entries[i].timestamp + entries[i-1].hash
    );
    if (entries[i].hash !== expected) {
      return { valid: false, brokenAt: entries[i].id };
    }
  }
  return { valid: true };
}
```

## 5. Data Governance

### 5.1 Data Classification

All data handled by this platform is classified as:

> **UNCLASSIFIED // PUBLIC** — Information obtained from publicly available sources that does not require security clearance to access.

### 5.2 Data Retention

| Data Type | Retention Period | Deletion Method |
|-----------|-----------------|-----------------|
| Audit logs | Indefinite (immutable) | Not deletable |
| Evidence records | User-controlled | Soft delete (status change) |
| Case files | User-controlled | Soft delete with audit entry |
| Search queries | Session only | Not persisted |
| API responses | Cache TTL (60s) | Automatic expiry |

### 5.3 Data Minimization

- No PII is collected beyond OAuth profile (name, avatar)
- Search queries are not logged to the database
- API responses are cached in-memory only, not persisted
- File uploads go to S3 with non-enumerable keys

## 6. Network Security

### 6.1 Transport

- All client-server communication over HTTPS (TLS 1.2+)
- tRPC uses Superjson serialization (no raw SQL exposure)
- CORS restricted to application origin
- CSP headers prevent XSS injection

### 6.2 API Security

- Rate limiting on all external API calls
- No API keys exposed to the client (server-side proxy)
- External API failures gracefully degraded (no crash propagation)

### 6.3 Input Validation

- All tRPC inputs validated with Zod schemas
- SQL injection prevented by Drizzle ORM parameterized queries
- XSS prevented by React's default escaping + CSP headers
- File uploads validated for type and size before S3 storage

## 8. Incident Response

### 8.1 Threat Model

| Threat | Mitigation |
|--------|-----------|
| Unauthorized access | OAuth + JWT + role-based access |
| Data tampering | Immutable audit log with hash chain |
| API abuse | Rate limiting + server-side proxy |
| XSS/injection | Zod validation + React escaping + CSP |
| Supply chain | Dependency auditing + lockfile integrity |
| Prompt injection | Pliny agent monitors all LLM inputs |

### 8.2 Response Procedures

1. **Detection**: Security Perimeter page monitors threats in real-time
2. **Containment**: Connector disconnect capability for compromised integrations
3. **Investigation**: Audit log provides complete action history
4. **Recovery**: Checkpoint/rollback system for platform state
5. **Reporting**: Intelligence report generator for incident documentation
